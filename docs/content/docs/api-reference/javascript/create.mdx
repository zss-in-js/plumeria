export type create = <const T extends Record<string, CreateStyleValue>>(rule: T)=> CreateReturnType<T>;

`create` defines css styles using plain JavaScript style objects.  
type-safe・autocomplete・lintable

Only the styles you use in `styleName` will be ondemand compiled.

## Example

```tsx title="TypeScript"
import * as css from '@plumeria/core'

const styles = css.create({
  container: {
    display: 'flex',
    justifyContent: 'space-between',
  },
  text: {
    fontSize: '14px',
    textDecoration: 'none',
  },
})

export default function App() {
  return (
    <div
      styleName={[
        styles.container,
        styles.text
      ]}
    />
  );
}
```

The call is cleaned up by the bundler.

**Compiled:**
```tsx
<div className="xxp6epoh xwbwwha1 xbief8v4 xzllkng3" />
```

**Generated CSS:**

```css
.xxp6epoh {
  display: flex;
}
.xwbwwha1 {
  justify-content: space-between;
}
.xbief8v4:not(#\#) {
  font-size: 14px;
}
.xzllkng3 {
  text-decoration: none;
}
```

Compiles each style CSS property into a unique, **atomic**, and **hashed** class name. This prevents style collisions and maximizes reusability.


## Dynamic values

You can use functions to define dynamic styles. These are processed as dynamic values at build time.

```tsx
const styles = css.create({
  palette: (color: string) => ({
    backgroundColor: color,
  }),
});

export const Color = ({ color }) => {
  return (
    <span styleName={styles.palette(color)}>Dynamic Color</span>
  )
}
```

## Variants bracket notation

You can resolve dynamic variant styles using standard bracket notation (e.g., `variantStyles[size]`). Plumeria's compiler automatically extracts these variants and creates static lookup tables at build time.

```tsx
import * as css from '@plumeria/core';

const styles = css.create({
  base: {
    display: 'inline-flex',
    borderRadius: '4px',
    backgroundColor: 'blue',
    color: 'white',
  }
});

const sizeStyles = css.create({
  small: {
    fontSize: '12px',
    padding: '4px 8px',
  },
  large: {
    fontSize: '18px',
    padding: '12px 24px',
  }
});

type Size = 'small' | 'large';
// can also use the type exhaustively via typeof sizeStyles

export const Button = ({ size }: { size: Size }) => {
  return (
    <button styleName={[styles.base, sizeStyles[size]]}>Button</button>
  );
};
```

<Callout type="warn" title="Function keys are ignored">
  Function keys (dynamic styles) within the style object are ignored during bracket notation lookup table compilation.
</Callout>

### Optimization Tip: Keep Common Properties in Base Styles

To keep the generated lookup table small and efficient, you should concentrate common, non-variant properties in your base styles (like `baseStyles`). 
In that case, a optimized cartesian product is not generated.

## Zero-Runtime logic
In Plumeria, all variants, dynamic props and conditional expressions are resolved at compile time by partially optimizing the Cartesian product and addition. There is absolutely no runtime overhead associated with merging objects or concatenating strings, as is common with many CSS-in-JS libraries.

### 1. Minimizing Combinations via Partial Cartesian Products
To avoid a combinatorial explosion, the Plumeria compiler constructs a **"property conflict graph"** between variants during static analysis. Independent variants—those that do not share conflicting properties (e.g., `size` and `theme` affecting entirely different CSS properties)—are treated as simple, independent mappings. Only variant axes that compete for the same properties (such as `margin` or `display`) are automatically grouped together, and only those specific combinations (partial decision tables) are expanded during the build process. Consequently, the number of combinations is constrained by the component's actual "local axes of conflict," preventing exponential growth unless variants are stacked haphazardly.

### 2. Constant $O(1)$ Runtime Cost
Even if the expansion causes the table to grow somewhat, this impact is limited to build time. At runtime, the browser performs a minimal operation: simply retrieving a single entry from a fully pre-computed, flat table in $O(1)$ time. This completely eliminates unnecessary JavaScript object allocations and the associated load on the Garbage Collector (GC).

<Callout title="Atomic static API">
  Unique atom classes are reused without duplication, keeping the total number of class names constant.  
  And all imports are stripped away by the compiler.
</Callout>
