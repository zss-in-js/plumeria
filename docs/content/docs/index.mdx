## Type-only
Typical CSS-in-JS solutions use several kilobytes in initial bundle size. This library completely replaces type calls and is complete with just type definitions and plugins.
Developers simply extend the type definition arguments as **schemas**, so no JS parsing is required and there are no memory allocation costs.


```json title="package.json"
{
  "name": "@plumeria/core",
  "type": "module",
  "exports": {
    ".": {
      "types": "./lib/css.d.ts", // API type definitions only
      "import": "./lib/css.js" // entry only empty export 0B by bundler
    },
  },
  "files": [
    "lib/"
  ],
}
```

## How It Works
The pre-build code is actually written in types and transformed by build plugins.  
Write styles with TypeScript autocomplete and type safety:
```tsx title="Before build"
import * as css from "@plumeria/core";

const styles = css.create({
  text: {
    fontSize: 12,
    color: 'navy',
  },
  size: {
    width: 120,
  },
});

<div className={css.use(styles.text, styles.size)} />
```

The build plugin transforms this into pure atomic CSS:
```tsx title="After build"
// All imports and style objects removed
<div className="xhrr6ses xvbwmxqp xhk51flp" />
```

## Type-only Advantage

By treating styles as **type schemas** instead of runtime objects:
- No JavaScript bundle
- No JavaScript runtime
- No JavaScript parsing overhead
- No memory allocation for style objects
- Tree-shaking removes all framework code
- Only atomic class names and optimized CSS remain in production

## Abstracting the Psychological Cost
I believe the core issues with CSS frameworks are reliability and transparency.
When actually building a UI, it’s sometimes impossible to know exactly how many kilobytes it’s consuming.

Software is like a building; you must always design with its longevity in mind.

- The psychological burden of postponing class conflict issues
- The psychological burden of continuing to write black-box code
- The psychological burden of constantly thinking about when to remove it
- The psychological burden of the potential for significant time consumption in the long run
- The engineering burden of performance considerations
- The psychological burden of layout breaks not manifesting as errors

These issues persist even when building with CSS.
If you are not accustomed to engineering, the cost of these psychological burdens can be greater than expected.

However, Zero-Runtime CSS libraries significantly reduce this problem.
This is because the costs mentioned earlier can be controlled.


## The Concept of Zero-Runtime
There is no such thing as a library where everything is truly Zero-Runtime.
Even pure CSS incurs a slight increase in compilation time due to the build pipeline of the bundler.
Zero-Runtime CSS-in-JS adds additional processing time for parsing and transformation. However, with a clear understanding of these factors, the term “Zero-Runtime” is often used to describe the concept that no runtime code remains in the execution environment.

- Strict CSS value validation and code formatting via linters
The target is no longer a black box, and sorting and values are automatically enforced.
This acts as a safety net, allowing engineers to catch errors before they occur and enabling automatic corrections.
As a result, a clean state is maintained both psychologically and in the code.

This is because, by its nature, CSS-in-JS can directly benefit from the features of TypeScript and JavaScript.

- Challenges Solved by Build Tools
The style definitions you write are compiled into CSS and then eliminated through the build pipeline.
This provides the reassurance that what you wrote does not remain, and that the output CSS is smaller than the original definitions.
As a result, your mental state during styling is kept clean by the architecture.

Plumeria is designed to manage these invisible burdens in a controlled manner.


## Usage
Use `style.create()` to define an object as a style.
At this point, the atomic hash is determined, and all that remains is to receive it via `style.use()`.

```tsx
import * as style from ‘@plumeria/core’;

export const Footer = () => {
  return <div className={style.use(styles.text)}>MyText</div>;
};

const styles = style.create({
  text: {
    fontSize: 16,
  },
});

```
As a best practice for coding conventions, it is recommended to place this code below the component.

Conditional Expressions, Ternary Operators, and Specificity Rules

`style.use()` accepts ternary operators and conditional expressions, and the style properties on the right side of the argument always take precedence.
For example, if `isActive` is true, the `color` of `active` will override the `color` of `base`.
```ts
style.use(styles.base, isActive && styles.active)
```
Note: Class name conflicts are strictly controlled by the compiler through merging performed by the `use` function’s static analysis.
